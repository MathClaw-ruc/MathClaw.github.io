把 MathClaw 部署到真实沟通场景里

MathClaw 的部署重点不是单独启动一个 Web 页面，而是让 gateway、console、memory 和 channel 在同一工作区里持续运转。这里给出接入方向、运行建议和上线检查项。

核心进程gateway + console

Channels

接入方式

企业微信

适合真实教学运营场景，连接稳定后可直接把学习代理投入日常使用。

QQ

适合作为补充接入方式，与企业微信共享相同的工作区和记忆链路。

飞书

适用于需要额外企业协作入口的场景，可与其他通道并行运行。

Web Console

既是管理入口，也是解题工作台的直接使用界面，适合演示与自测。

Checklist

上线检查项

1. 配置可读

  • 确认 `~/.mathclaw/config.json` 或对应环境变量可被读取。
  • 模型与 provider 路径不要留远端硬编码。
  • console 启动时明确指定 workspace。

2. 工作区可写

  • `workspace/sessions` 可写入原始对话。
  • `workspace/memory` 可写入 daily / weekly memory。
  • `workspace/cron/jobs.json` 可更新心跳状态。

3. 页面可见

  • 运行状态页返回 200。
  • 记忆页中的知识图谱和错题图谱可以加载。
  • Skills 与心跳页能读到当前状态。

4. 通道已认证

  • 企业微信通道完成认证。
  • QQ / 飞书按实际场景启用。
  • 只保留需要的通道，避免无效进程常驻。
FAQ

部署说明

控制台和网关为什么要分开启动？

控制台是管理和使用入口，网关负责消息循环与通道接入。分开运行可以避免前端重启时打断通道。

为什么强调 workspace？

因为 sessions、memory、graphs 和 cron 都依赖统一工作区。没有它，记忆、图谱和心跳就无法稳定衔接。

如何确认系统仍然健康？

优先看运行状态页和心跳页；前者看 gateway / console / channel，后者看日总结和周总结是否按计划运行。

Skills 应该什么时候启用？

仅在附件回复场景下启用。它们不会替代正常回答，而是追加一种更贴合教学目的的额外输出框。
