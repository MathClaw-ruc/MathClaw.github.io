Deploy

Run MathClaw in a real communication environment

Deployment is not only about serving a web page. The real goal is to keep gateway, console, memory, and channels running on top of one shared workspace. This page outlines connection options, operating suggestions, and launch checks.

Channels

Connection surfaces

WeCom

Best suited for real teaching operations and daily deployment once authentication is stable.

QQ

A complementary entry point that shares the same workspace and memory flow as the primary channels.

Feishu

Useful when an extra enterprise collaboration channel is needed alongside other deployments.

Web Console

Works both as the management entry point and the direct solver workspace for demos and internal testing.

Checklist

Launch checklist

1. Config is readable

  • Verify `~/.mathclaw/config.json` or the matching environment variables are available.
  • Do not keep machine-specific absolute paths in model config.
  • Always pass the workspace explicitly when starting the console.

2. Workspace is writable

  • `workspace/sessions` must accept raw conversation writes.
  • `workspace/memory` must accept daily and weekly memory output.
  • `workspace/cron/jobs.json` must update heartbeat status.

3. Pages are healthy

  • The runtime page should return 200.
  • The memory page should load both graphs successfully.
  • Heartbeat and Skills should show current state instead of empty placeholders.

4. Channels are authenticated

  • WeCom should complete authentication cleanly.
  • Enable QQ and Feishu only when they are actually used.
  • Keep the running set of channels minimal to reduce idle processes.
FAQ

Operator notes

Why separate gateway and console?

The console is the control and usage surface, while the gateway owns channel IO and agent execution. Running them separately avoids unnecessary restarts.

Why is the workspace so important?

Sessions, memory, graphs, and cron state all rely on one shared workspace. Without it, planning, graphs, and heartbeat cannot stay connected.

How do I verify runtime health?

Start with the Runtime Status page and the Heartbeat page. The first verifies the service layer; the second verifies scheduled summary cadence.

When should Skills be enabled?

Only for attachment reply scenarios. Skills do not replace the default answer; they append an extra structured teaching style after it.
